# ADR-003: Repository Pattern for Storage Abstraction

**Date:** Day 28
**Status:** Accepted
**Decider:** Udit Rawat

## Context

Business logic (`services.py`, `commands.py`) was calling `json_store.py` directly. Changing to SQLite would require modifying every caller.

## Decision

Introduce a `TaskRepository` abstract base class with a concrete `JsonTaskRepository`. All business logic talks to the interface — never to `json_store.py` directly.

## Rationale

- **Swappable backends** — changing from JSON to SQLite requires changing one line (the backend name in config)
- **Testable** — `InMemoryTaskRepository` can be used in tests with no file I/O
- **Principled** — enforces the Dependency Inversion Principle
- **Phase 3 ready** — `SqliteTaskRepository` and `PostgresTaskRepository` can be added without touching services

## Consequences

- Added `taskflow/repositories/` subpackage (3 files)
- `get_repository()` singleton via `@lru_cache` replaces direct storage imports in services
- Slight indirection added to every data access call
