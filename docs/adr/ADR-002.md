# ADR-002: Use Task Class Hierarchy Instead of Plain Dicts

**Date:** Day 12
**Status:** Accepted
**Decider:** Udit Rawat

## Context

Tasks were stored as plain dicts from Day 05. As the codebase grew, this led to scattered validation, string-comparison bugs, and no ability to attach behaviour to task objects.

## Decision

Replace all task dicts with a `Task` class hierarchy:
`Task` → `UrgentTask`, `RecurringTask`, `DeadlineTask`

## Rationale

- **Encapsulation** — validation in `__init__` and `@property.setter`; invalid state impossible to construct
- **Polymorphism** — `mark_done()` behaves differently per subclass; callers don't need to know which type
- **Sortability** — `__lt__` enables `sorted(tasks)` with no key argument
- **Hashability** — `__hash__` enables use in sets and as dict keys
- **Round-trip serialisation** — `to_dict()` / `from_dict()` on every class keeps JSON storage clean
- **Self-description** — `__str__` / `__repr__` give human and developer views

## Rejected Alternatives

- **TypedDict** — validates only at type-check time; no runtime guards, no methods
- **dataclasses** — possible but lacks property validation and clean method overriding
- **Pydantic** — excellent choice; deferred to Day 33 where it is introduced properly

## Consequences

- Slightly more verbose construction than dict literals
- `from_dict()` must be updated when new subclass attributes are added
- Required `TaskFactory.create_from_dict()` for polymorphic deserialisation
